# Production Agent Deployment — Reference Architecture

This is the canonical example of how to deploy trustcard with an AI agent
that calls MCP servers. It shows the mental model: where the proxy sits,
what the manifest does, where policies are defined, and where receipts go.

## Architecture

```
  ┌─────────┐
  │  Agent   │  (Claude, Devin, custom agent, etc.)
  │          │
  └────┬─────┘
       │ JSON-RPC over stdio
       │
  ┌────▼──────────────────────────────────────────────┐
  │  mcp-proxy                                         │
  │                                                    │
  │  ┌──────────────┐  ┌──────────────┐  ┌─────────┐ │
  │  │  Manifest     │  │  Pin Store   │  │ Receipts│ │
  │  │  (approved    │  │  (TOFU state │  │ (signed │ │
  │  │   tool set)   │  │   per server)│  │  chain) │ │
  │  └──────────────┘  └──────────────┘  └─────────┘ │
  │                                                    │
  │  Gate 1: trust-state continuity (manifest vs live) │
  │  Gate 2: invocation policy (per-call authorization)│
  └────┬───────────────────────────────────────────────┘
       │ JSON-RPC over stdio (filtered)
       │
  ┌────▼─────────┐
  │  MCP Server   │  (filesystem, github, database, etc.)
  │               │
  └───────────────┘
```

## Components

### 1. Manifest (`manifest.json`)

Generated by `mcp-trustcard gen-manifest`. Contains the approved tool set,
schema hashes, danger analysis, and an expiry date. The proxy blocks calls
to tools not in the manifest, and strips unapproved tools from `tools/list`.

```bash
# Generate a manifest for a local server (e.g. MAOS)
mcp-trustcard gen-manifest \
  --save-manifest manifest.json \
  --allow-tool maos_verify_command \
  --expires-in 90 \
  -- uv run maos mcp serve

# Inspect it
mcp-trustcard inspect manifest.json
```

### 2. Pin Store (`pins.json`)

TOFU state: the first time you connect to a server, its identity is pinned.
Every subsequent connection must match or produce an auditable diff. This is
the client's ground truth — not the server, not the registry.

```bash
# Pin a server after first verification
mcp-trustcard pin -- <server-cmd>

# List all pins
mcp-trustcard pins

# Inspect the pin store
mcp-trustcard inspect pins.json
```

### 3. Proxy (`mcp-proxy`)

Sits between the agent and the MCP server. Intercepts `tools/list` (strips
unapproved/dangerous tools) and `tools/call` (blocks unauthorized calls).
All other JSON-RPC methods pass through untouched.

```bash
# Start the agent through the proxy
mcp-proxy --manifest manifest.json -- <server-cmd>

# Strict mode: exit on any drift (CI/production)
mcp-proxy --manifest manifest.json --strict -- <server-cmd>
```

### 4. Policy (Gate 2)

Defined in application code at Guard construction time. Not a file, not a
DSL — composable rule predicates. This is deliberate: a policy language
would force global agreement and destroy the per-relying-party model.

```javascript
import { InvocationPolicy, denyTools, constrainArg, requireApprovalForDestructive } from "trustcard/lib/policy.js";
import { Guard } from "trustcard/lib/guard.js";

const policy = new InvocationPolicy({
  rules: [
    // Deny specific tools entirely
    denyTools(["filesystem.delete"]),

    // Require human approval for destructive tools
    requireApprovalForDestructive(),

    // Constrain a tool's arguments
    constrainArg("filesystem.read", "path", (p) => p.startsWith("/data/")),
  ],
});

const guard = new Guard({
  mode: "enforce",
  invocationPolicy: policy,
  relyingParty: "my-agent",
  environment: "production",
  receiptKey: agentPrivateKey,  // optional: enables signed receipts
});
```

### 5. Receipts (`receipts.jsonl`)

Every call produces a receipt binding the exact contract version to the
call. With a receipt key, receipts are Ed25519-signed and hash-chained.
Receipts are the non-repudiation layer: they prove *what was authorized
and called*, not what the tool *did*.

```javascript
import { makeReceiptSink, verifyReceiptChain, loadReceipts } from "trustcard/lib/receipts.js";

const sink = makeReceiptSink("receipts.jsonl");
const guard = new Guard({ receiptSink: sink, receiptKey: agentPrivateKey });

// Later: verify the receipt chain
const receipts = loadReceipts("receipts.jsonl");
const result = verifyReceiptChain(receipts);
assert(result.ok, `chain broken: ${result.errors.join(", ")}`);
```

## Deployment steps

1. **Generate the manifest** for each MCP server your agent will call.
2. **Review the danger analysis** — use `inspect` to see which tools are
   flagged. Use `--allow-tool` for reviewed tools with their own safety
   constraints.
3. **Define Gate 2 policies** in your agent's Guard construction.
4. **Start the agent through the proxy** — one proxy per server.
5. **Collect receipts** to a file or log sink for audit.
6. **Set up manifest regeneration** — manifests expire (default 90 days).
   Re-run `gen-manifest` before expiry to keep the danger analysis fresh.

## What this deployment does NOT do

- **No shared state across proxy instances.** Each agent has its own pin
  store, its own manifest, its own receipts. A server REVOKED by agent A
  is still PINNED for agent B.
- **No runtime sandboxing.** trustcard enforces the *contract*, not the
  *behavior*. A signed read-only tool can still behave badly. Use
  OS-level isolation for runtime containment.
- **No automatic key rotation.** Key rotation is break-glass (old key signs
  new). Distribute rotation certificates out-of-band.
- **No registry lookup.** The pin store is local TOFU state. There is no
  central registry to check whether a publisher key has been revoked.
